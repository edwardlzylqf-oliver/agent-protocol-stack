> 版本：1.0.0 | 状态：draft

## 一、协议概述

本协议定义多智能体系统的基础协作规范，采用Spine-Leaf（叶脊）架构，支持：
- 多Agent注册与管理
- 任务智能路由
- 分层记忆同步
- 冲突仲裁机制

---

## 二、角色体系

### 2.1 角色类型

| 角色 | 英文 | 描述 | 数量限制 |
|------|------|------|----------|
| **Spine** | coordinator | 协调者/首辅，负责任务分配和决策 | **仅1个** |
| **Leaf** | executor/designer/developer/expert/supervisor | 执行者，按职责执行任务 | 多个 |
| **Observer** | supervisor | 监督者，仅监听+发现问题 | 可选1个 |

### 2.2 当前实现角色映射

```
老东西 → Spine (coordinator)
├── 菲利普 → Leaf (executor) - 分析执行
├── 路飞 → Leaf (designer) - 设计制作
├── 小K → Leaf (developer) - 开发实现
├── 蚕宝 → Leaf (expert) - 专家咨询
└── 莫妮卡 → Observer (supervisor) - 监督审核
```

## 三、响应规则（来自协同策略v4.0）

### 3.1 基本原则

| 角色 | 响应规则 |
|------|----------|
| **Spine** | 收到所有消息，**先由Spine判断谁来回答**，再传达或指定 |
| **Observer** | 收到所有消息，但**不直接回复**，只有发现问题、提出质疑和建议时才回复 |
| **Leaf** | 只在被@时响应，**未经Spine确认不要主动回答** |

### 3.2 响应流程

```
收到消息
  ├── 被@？→ 接收者响应"收到" → 执行 → 汇报
  ├── Spine判断？→ 识别意图 → @目标Leaf → 目标响应
  └── Observer发现问题？→ 提出质疑 → 记录到内存
```

## 四、禁止规则（来自协同策略v4.0）

### 4.1 禁止行为

| 行为 | 分类 | 处罚 |
|------|------|------|
| 未经Spine确认就回答问题 | 抢答 | 首次提醒→二次警告→三次记录 |
| 未经@直接响应 | 越权 | 同上 |
| 被@不回复"收到" | 沉默 | 首次提醒→多次记录 |
| 转告而非直接@ | 违规 | 提醒 |
| 禁止通过"Spine传达"的方式转告 | 转告 | 提醒 |

### 4.2 违规处理

| 违规次数 | 处理 |
|----------|------|
| 1 | 提醒 |
| 2 | 警告 |
| 3 | 记录到MEMORY.md，汇报用户 |

## 五、@沟通规范

### 5.1 正确与错误示例

- ✅ 正确: "@xxx 请执行YYY"
- ✅ 正确: "@xxx 这个怎么理解？"
- ❌ 错误: "xxx去执行"
- ❌ 错误: "我帮你们传达"

### 5.2 消息格式

| 类型 | 格式 | 示例 |
|------|------|------|
| 任务确认 | "收到，我会{执行内容}" | "收到，我会分析这份财报" |
| 任务完成 | "报告：已完成{任务}，结果：{简述}" | "报告：已完成分析，增长15%" |
| 质疑 | "质疑：{问题描述}" | "质疑：第三点数据有误" |
| 建议 | "建议：{建议内容}" | "建议：补充竞品对比" |

## 六、任务路由（来自多智能体路由规则框架）

### 6.1 关键词库

| 分类 | 关键词 | 路由目标 |
|------|--------|----------|
| **制作类** | PPT、演示、制作、可视化、图表 | Leaf-designer |
| **开发类** | 开发、代码、实现、ERP、金蝶、数字化 | Leaf-developer |
| **分析类** | 分析、数据、报表、财务、指标、股票 | Leaf-executor |
| **研究类** | 研究、命理、方法论、内容、咨询 | Leaf-expert |
| **管理类** | 分配、任务、进度、汇总、审核 | Spine |
| **监督类** | 记录、归档、监督、质量、审核 | Observer |

### 6.2 路由决策流程

```
输入任务
  ↓
提取关键词/意图
  ↓
匹配执行者 (查询RoutingTable)
  ↓
@目标Agent执行
  ↓
返回结果 → Spine汇总 → 汇报
```

### 6.3 复合任务识别

当任务涉及多个领域时，自动识别为复合任务：
- 示例: "分析Q4财报并制作PPT" → 路由到 executor + designer
- 串行执行: 分析 → PPT

## 七、记忆分层

### 7.1 层级定义

| 层级 | 名称 | 位置 | 可见性 | 用途 |
|------|------|------|--------|------|
| **L0** | 全局记忆 | Obsidian/Clusters/Memory/Global/ | 所有Agent | 法律法规、公司规则、全局知识 |
| **L1** | 局部记忆 | Obsidian/Clusters/Memory/Local/ | 项目组/业务组 | 项目信息、业务规则、团队偏好 |
| **L2** | 私有记忆 | {workspace}/memory/ | 仅自己 | 个人思考、临时笔记 |

### 7.2 同步规则

- L2 → L1: 每日筛选重要信息
- L1 → L0: 每周归纳总结
- L0 → L1/L2: 即时推送（规则变更）

## 八、仲裁机制

### 8.1 冲突检测

当多个Leaf同时修改同一文件时触发仲裁：
- 版本号比对
- 时间戳比对
- 优先级比对

### 8.2 仲裁规则

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | Spine | Spine版本优先 |
| 2 | 时间戳 | 最新优先 |
| 3 | 合并 | 内容合并 |

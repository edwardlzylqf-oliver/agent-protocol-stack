## 一、任务生命周期

```
┌─────────────────────────────────────────────────────────────────┐
│                      任务状态流转                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  created ──→ assigned ──→ executing ──→ completed              │
│      │            │              │              │                │
│      │            │              │              ↓                │
│      │            │              └──────────────→ failed         │
│      │            │                                       │       │
│      │            └─────────────────────────────────────┘       │
│      │                                                  │       │
│      └──────────────────────────────────────────────────┘       │
│                     (可重试恢复)                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 状态定义

| 状态 | 描述 | 触发条件 |
|------|------|----------|
| **created** | 已创建 | 用户提交任务 |
| **assigned** | 已分配 | Spine@Agent确认 |
| **executing** | 执行中 | Agent开始处理 |
| **completed** | 已完成 | 结果返回 |
| **failed** | 失败 | 执行异常/超时 |

## 二、任务类型

### 2.1 单一任务

简单任务，由单个Agent完成：
- "分析这份数据" → 路由到 leaf_001
- "制作一个PPT" → 路由到 leaf_002

### 2.2 复合任务

复杂任务，需多个Agent协作：
- "分析数据并制作PPT" → 需 leaf_001 + leaf_002
- 执行顺序：先分析 → 后PPT

### 2.3 批量任务

同类型任务批量处理：
- "分析这10份财报" → 队列处理

## 四、任务流转流程

### 4.1 任务创建

1. 用户提交任务（通过飞书/web/API）
2. Spine接收并解析任务
3. 判断任务类型（单一/复合/批量）

### 4.2 任务分配

1. Spine查询RoutingTable
2. 确定目标Agent
3. @目标Agent发送任务
4. 等待Agent确认"收到"

### 4.3 任务执行

1. Agent开始处理
2. 如需长时间，定期汇报进度
3. 完成后返回结果

### 4.4 任务完成

1. Spine汇总结果
2. 同步到记忆系统
3. 汇报给用户

## 五、复合任务处理

### 5.1 识别

当任务涉及多个技能领域时自动识别：
```
"分析数据并制作PPT" 
  → 包含: 分析技能 + PPT技能
  → 识别为: 复合任务
```

### 5.2 拆分

复合任务拆分为子任务：
```json
{
  "task_id": "task_composite_001",
  "type": "composite",
  "sub_tasks": [
    {
      "sub_id": "task_001",
      "skill": "分析",
      "agent": "leaf_001",
      "status": "pending"
    },
    {
      "sub_id": "task_002", 
      "skill": "PPT",
      "agent": "leaf_002",
      "status": "pending",
      "depends_on": ["task_001"]  // 依赖上一任务
    }
  ]
}
```

### 5.3 执行顺序

- **串行**: A完成 → B开始
- **并行**: A和B同时开始（无依赖时）

## 六、容错机制

### 6.1 重试策略

| 失败类型 | 重试次数 | 间隔 |
|----------|----------|------|
| 网络错误 | 3次 | 10s, 30s, 60s |
| 执行超时 | 2次 | 60s, 120s |
| 资源不足 | 2次 | 30s, 60s |

### 6.2 失败处理

1. 重试达到上限 → 标记为failed
2. 记录失败原因
3. 通知Spine重新分配
4. 如无法分配 → 标记需人工介入

### 6.3 任务超时

- 默认超时: 30分钟
- 可自定义: `deadline`字段
- 超时处理: 重试或释放资源

---

## 七、任务优先级

| 优先级 | 描述 | 处理方式 |
|--------|------|----------|
| **urgent** | 紧急 | 立即处理，插队最前 |
| **high** | 高 | 优先处理 |
| **normal** | 普通 | 按顺序处理 |
| **low** | 低 | 空闲时处理 |
